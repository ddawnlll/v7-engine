# Codebase Map: cross-domain contracts + runtime integration (PolicyCritic advisory wiring target)

## Subsystem Identity
The runtime orchestrates a scan -> decision -> readiness transition across two layers: the live V4-era scan loop (runtime/) and the V7 contract-family flow (v7/docs/contracts + v7/docs/runtime/runtime_integration.md).

LIVE SCAN LOOP (runtime/docs/ai_summary.md R.2, R.3): A request arrives at POST /api/v3/scans or AutonomousLoop.run_once() carrying symbols, intervals, modes, requested_by. ScanRuntime (runtime/scan_runtime.py) creates a scan_run row (status=RUNNING), expands scope via _normalize_mode_intervals and UniverseFilterService symbol throttling, fetches market snapshots (ThreadPoolExecutor), then calls the analyzer wrapper per (symbol, interval, mode). The analyzer (services/analyzer_core.py) runs a layered pipeline: circuit-breaker pre-check -> regime -> trend -> structure -> entry confirmation -> probability model -> stop model -> timing model -> execution-quality penalties -> learning adjustments -> FINAL GATING (confidence, RR, EV; failure -> NEUTRAL with decision_path.neutral_stage). It returns BUY/SELL/NEUTRAL plus decision_path and audit_json. Non-neutral eligible signals create paper orders via paper_execution.py. Scan control (scan_control.py) checks cooperative pause/stop between tasks.

V7 CONTRACT FLOW (runtime_integration.md): Market/Data State -> Request Builder -> AnalysisRequest (carries requested_trade_mode/model_scope) -> Mode Router (TradeModeRouter in runtime/runtime/runtime_router.py resolves a per-mode champion ModelArtifact from ModelRegistry; falls back to global champion or v4_default) -> Scope-Compatible Artifact Selection -> Engine -> AnalysisResult -> Result Validator -> Runtime Interpretation -> DecisionEvent -> Execution Eligibility -> Execution/No-Execution -> TradeOutcome. Runtime owns orchestration/persistence/safety/lifecycle; engine owns scores/confidence/expected_r/recommended_action/timing.

PER-MODE READINESS STATES (runtime_integration.md): Each mode (SWING, SCALP, AGGRESSIVE_SCALP) has an independent one-way readiness state: DESIGN_LOCKED -> RESEARCH -> BACKTEST_ELIGIBLE -> SHADOW_ELIGIBLE -> PAPER_ELIGIBLE -> TINY_LIVE_ELIGIBLE -> LIVE_ELIGIBLE, plus DISABLED (enterable from any state on kill-switch). State gates what the mode may do: RESEARCH returns NO_TRADE (fallback_used=research_only); SHADOW emits actionable decisions but blocks execution; PAPER routes to paper simulation; TINY_LIVE/LIVE execute with kill switches. Runtime routes only to artifacts whose readiness allows the requested environment; a DISABLED mode returns NO_TRADE with degradation reason (no silent fallback to another mode). Promotion G0..G10 gates (v7/docs/pipeline/evaluation.md) advance a mode to the next state.

EXECUTION ELIGIBILITY STACK (runtime_integration.md): runtime moves away from 'confidence >= min => execute' toward a layered gate: (1) structural validity, (2) engine actionability (is_actionable==true, recommended_action in {LONG_NOW,SHORT_NOW}), (3) confidence gate, (4) economic gate (expected_r >= min, optional expected_drawdown <= max), (5) timing gate (advisory-first: inspect entry_readiness/entry_valid_for_bars; later optional hard gate on MISSED/CHASING), (6) operational hard gate (exchange healthy, no duplicate, cooldown, account/risk controls, kill switches). A result may be economically actionable but operationally not executable; runtime records that distinction in DecisionEvent.runtime_interpretation.

POLICY STAGE (v7/docs/ai_summary.md sec 19, docs/pipeline/policy.md): Policy is where learned evidence becomes a decision. Directional action must pass all decision gates: probability/confidence, no-trade comparison, decision margin, expected-R, cost-adjusted expectancy, adverse-pressure/drawdown, regime consistency, degradation/fallback. Failure -> NO_TRADE or degraded-safe. Stage order first-phase: Policy -> Portfolio -> Risk -> Runtime execution eligibility (sec 21.2).

## Critic Integration Points
A PolicyCritic is an ADVISORY component. Per CLAUDE.md domain boundaries, V7 owns final trade decisions (policy acceptance); runtime owns execution eligibility; the critic must NOT own the final trade decision. So the critic emits a PolicyCriticReview that runtime/V7 policy CONSUMES but does not blindly obey — it translates advisory verdicts into existing policy/runtime_interpretation fields.

WHERE IT PLUGS IN: Between the hard policy gates (v7/docs/ai_summary.md sec 19.2: confidence, no-trade, margin, expected-R, cost-adjusted expectancy, adverse-pressure, regime, degradation) and the final policy decision / portfolio stage (sec 20). Stage order is Policy -> Portfolio -> Risk -> Runtime execution eligibility (sec 21.2). The critic sits inside the Policy stage as a post-gate advisory review: after a candidate passes the hard gates, the critic reviews the gated candidate and emits ALLOW / DOWNWEIGHT_CONFIDENCE / VETO_TO_NO_TRADE / REQUIRE_REVIEW. This is strictly between hard gate and final gate (the final gate being the operational hard gate in the execution eligibility stack, step 6).

CONTRACT REGISTRATION (per contracts/README.md 'How to Add a New Contract'): (1) Define contracts/schemas/policy_critic_review.schema.json (owner_domain: v7, since V7 owns policy acceptance; producers: v7 [or a v7-registered critic service], consumers: v7, runtime). Required fields: schema_version, review_id, decision_event_id (or request_id), verdict enum [ALLOW, DOWNWEIGHT_CONFIDENCE, VETO_TO_NO_TRADE, REQUIRE_REVIEW], confidence_adjustment_factor (for DOWNWEIGHT), rationale, review_tags, critic_artifact_version, is_advisory=true. (2) Add entry to contracts/registry.json. (3) Add compatibility rule in compatibility.json if runtime consumes it cross-domain (source_contract: PolicyCriticReview, target_contract: DecisionEvent or TradeOutcome). (4) Add field mapping in contracts/mappings/ (policy_critic_to_decision_event.json/md) showing how verdict maps to runtime_interpretation fields. (5) Add minimal fixture in contracts/fixtures/. (6) Add integration test in integration/tests/ (schema_parity + fixture_roundtrip).

RUNTIME WIRING: Register the critic as a service mirroring AnalyzerEngineRegistryService — a PolicyCriticRegistryService registering critic instances satisfying a PolicyCritic Protocol (review(request, result, policy_state) -> PolicyCriticReview), selected via a setting (e.g. POLICY_CRITIC_ACTIVE). The critic is invoked from the analyzer_engine_adapter.py normalization boundary OR from a new policy stage step in scan_runtime after the analyzer returns and before execution_orchestrator. The verdict is consumed as follows, all into EXISTING DecisionEvent.runtime_interpretation fields (no new execution semantics):
- ALLOW -> policy_passed stays true, no change.
- DOWNWEIGHT_CONFIDENCE -> multiply scores.confidence by confidence_adjustment_factor; record in observability/suppression_reason='critic_downweighted'; may re-trip the confidence gate.
- VETO_TO_NO_TRADE -> policy converts recommended_action to NO_TRADE; policy_passed=false; suppression_reason='critic_veto'; runtime_safe_action='NO_TRADE'. This is an advisory veto made effective BY policy, not by the critic directly — preserving 'V7 owns final trade decisions'.
- REQUIRE_REVIEW -> should_surface_to_review=true; review_tags appended; execution may still proceed unless config requires review-block.

The critic verdict must be persisted in DecisionEvent (runtime_interpretation) and echoed to TradeOutcome lineage for offline evaluation. It must NOT call the broker, NOT bypass simulation costs, NOT rewrite engine scores directly (it adjusts confidence via a recorded factor that policy applies).

Because AnalysisRequest/AnalysisResult/DecisionEvent are not yet in registry.json (they live as V7-local docs + runtime Pydantic models), the cleanest path is to register PolicyCriticReview as a V7-owned contract in registry.json and add a compatibility pair PolicyCriticReview<->DecisionEvent (or PolicyCriticReview<->TradeOutcome) in compatibility.json, establishing the critic as a first-class advisory artifact in the contract family.

## Available Reward Signals for Offline RL Training
The runtime/contract family exposes a rich offline reward surface for RL training of a PolicyCritic, all from TradeOutcome (contracts/schemas/trade_outcome.schema.json) and SimulationOutput:
- TradeOutcome.realized_r (realized net R from best action)
- TradeOutcome.regret_r (regret vs counterfactual best action)
- TradeOutcome.mfe_r / mae_r (max favorable / adverse excursion in R)
- TradeOutcome.saved_loss_score (from SimulationOutput.no_trade_outcome — value of NOT trading)
- TradeOutcome.missed_opportunity_score (cost of NO_TRADE when a directional action would have won)
- TradeOutcome.path_quality_score
- TradeOutcome.is_good_decision (boolean label given outcome)
- TradeOutcome.best_realized_action / counterfactual_best_action (LONG_NOW/SHORT_NOW/NO_TRADE)
- TradeOutcome.outcome_status, resolution_reason, exit_reason (terminal state signal)
- DecisionEvent.decision_summary.confidence / expected_r / decision_margin / long_score / short_score / no_trade_score (the pre-outcome state features for the critic)
- DecisionEvent.runtime_interpretation.policy_passed / portfolio_blocked / risk_blocked / suppression_reason (whether the critic's verdict aligned with later outcome)
- SimulationOutput (contracts/schemas/simulation_output.schema.json) comparative economics for counterfactual replay.

For offline RL, a critic training loop would pair (DecisionEvent state at decision time) x (PolicyCriticReview verdict emitted) -> reward from later TradeOutcome.realized_r / regret_r / is_good_decision, with saved_loss_score and missed_opportunity_score shaping NO_TRADE vs directional trade credit. The runtime does not emit a scalar live reward; rewards are constructed offline from the persisted event/outcome pair.

## Domain Boundary Constraints
From CLAUDE.md and v7/docs/ai_summary.md, the PolicyCritic MUST NOT:
- Own the final trade decision. V7 owns policy acceptance; the critic is ADVISORY. VETO_TO_NO_TRADE is a recommendation that V7 policy enacts, not a direct order the critic executes.
- Override risk gates or operational hard gates. 'Model confidence and expected R cannot override operational limits' (sec 21.3); 'Runtime does NOT override policy with model confidence' (CLAUDE.md). The critic cannot force execution past risk/portfolio blocks.
- Bypass simulation costs. 'Simulation owns economic truth. No component bypasses simulation costs.' The critic must not fabricate economic quality; it consumes SimulationOutput/expected_r as input.
- Emit broker orders or own execution. Engine/critic must not own broker order submission, persistence schemas, trade outcome materialization, event creation, execution ledgers (runtime_integration.md boundary rules).
- Silently rewrite the engine decision. 'Runtime must not own: re-deriving model scores, silently rewriting the engine decision, inventing hidden action semantics' (runtime_integration.md). The critic's DOWNWEIGHT must be a recorded multiplier, not a silent score change.
- Introduce a hidden veto. 'No hidden deterministic veto. No hidden suppression of learned opportunity. Deterministic influence visible and reviewable' (v7 ai_summary). Every critic verdict must be visible in DecisionEvent.runtime_interpretation with suppression_reason and review_tags.
- Add new trading modes or actions. CLAUDE.md forbidden actions. The critic verdicts map onto existing recommended_action values (NO_TRADE) and existing confidence adjustment, not new action enums.
- Be consumed directly by the interface. 'Interface observes runtime API only.' The React UI may surface critic verdicts via existing /api/v3 routes (e.g. review surfaces), but must not call the critic directly or switch behavior by critic generation.
- Violate import boundaries: lib/ must not import v7/alphaforge/simulation; a v7-owned critic service lives under v7/ or runtime/services/, not lib/.

The critic MUST: remain advisory-first (like the timing extension rollout), be config-gated (enable_policy_critic, policy_critic_mode), persist its verdict for review, and degrade safely (if critic unavailable, policy proceeds unchanged — like self-learning layer: 'If self-learning unavailable, V4 continues unchanged').

## Notes
Key finding: AnalysisRequest, AnalysisResult, and DecisionEvent are NOT yet registered in contracts/registry.json — they exist only as V7-local docs (v7/docs/contracts/*.md) and as runtime Pydantic models (runtime/services/analyzer_engine_contract.py, schema_version analysis_request.v1 / analysis_result.v1). Only TradeOutcome has a registered cross-domain JSON schema. So there are two integration options for PolicyCriticReview: (A) register it as a standalone V7-owned contract in registry.json with a compatibility pair to DecisionEvent/TradeOutcome — cleanest, follows the contracts/README.md procedure; (B) add it as a new section of the DecisionEvent runtime_interpretation without a separate contract. Option A is preferred because it makes the critic versioned, fixture-testable, and machine-auditable per the ACCP/contract discipline.

The critic mirrors two existing patterns: (1) the AnalyzerEngineRegistryService registration pattern (register instances satisfying a Protocol, select via setting, fallback safe) and (2) the self-learning/learning layer pattern (additive decision-correction layer that does NOT replace the base analyzer, defaults to bypass when unavailable, persists audit trail). The timing-extension rollout (advisory-first Phase 1 validate/persist/log, Phase 2 observability, Phase 3 optional gating) is the exact rollout template the critic should follow.

Exact insertion point in the pipeline: v7/docs/ai_summary.md sec 19.2 decision gates (hard policy gates) -> PolicyCritic review (advisory) -> sec 19.3 tie-break / final policy selection -> sec 20 Portfolio -> sec 21 Risk -> runtime execution eligibility stack step 6 (operational hard gate). The critic's VETO_TO_NO_TRADE is enacted by policy (sec 19) before portfolio/risk, preserving 'V7 owns policy acceptance' and 'no hidden veto' (verdict visible in runtime_interpretation.suppression_reason).

No live scalar reward is emitted by runtime; offline RL rewards are constructed from persisted (DecisionEvent, TradeOutcome) pairs using realized_r/regret_r/is_good_decision/saved_loss_score/missed_opportunity_score.
